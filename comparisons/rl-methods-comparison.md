# RL 对齐方法对比

## 对比概览

| 方法 | 数据需求 | 奖励模型 | 训练稳定性 | 实现复杂度 |
|------|----------|----------|------------|------------|
| [[rlhf\|RLHF]]/PPO | 成对偏好 | ✅ 需要 | 低 | 高 |
| [[dpo\|DPO]] | 成对偏好 | ❌ | **高** | **低** |
| [[kto\|KTO]] | 二元标注 | ❌ | 高 | 低 |
| [[grpo\|GRPO]] | 无标注 | ❌ (规则) | 中高 | 中 |
| ORPO | 成对偏好 | ❌ | 高 | 低 |

## 详细对比

### 数据格式需求

| 方法 | 数据格式 | 获取成本 | 数据量需求 |
|------|----------|----------|------------|
| RLHF/PPO | 成对偏好 + RM数据 | **最高** | 10K-100K 偏好 + 100K+ RM |
| DPO | 成对偏好 (chosen/rejected) | 高 | 5K-50K |
| KTO | 二元标注 (好/坏) | **低** | 5K-50K |
| GRPO | 仅提示 (规则评分) | **最低** | 1K-10K |
| ORPO | 成对偏好 | 高 | 5K-50K |

```json
// RLHF: 需要排序数据 + 训练奖励模型
{"prompt": "...", "chosen": "...", "rejected": "..."}

// DPO: 需要成对偏好
{"prompt": "...", "chosen": "...", "rejected": "..."}

// KTO: 只需要好/坏标签
{"prompt": "...", "completion": "...", "label": true}

// GRPO: 只需要提示
{"prompt": "..."}  // + 规则奖励函数
```

### 训练流程复杂度

| 方法 | 步骤 | 模型数量 | 总训练时间 |
|------|------|----------|------------|
| RLHF | SFT → RM → PPO | **3个** | **最长** |
| DPO | SFT → DPO | 2个 | 中等 |
| KTO | SFT → KTO | 2个 | 中等 |
| GRPO | 直接 GRPO | **1个** | **最短** |
| ORPO | SFT + ORPO | 2个 | 短 |

### 计算资源需求

| 方法 | GPU 显存 (7B) | 训练速度 | 额外开销 |
|------|--------------|----------|----------|
| RLHF/PPO | 40GB+ (3模型) | **最慢** | RM 训练 + KL 约束 |
| DPO | 20GB | 快 | 参考模型前向 |
| KTO | 20GB | 快 | 参考模型前向 |
| GRPO | 24GB | 中 | 多次生成 (K个回答) |
| ORPO | 20GB | **最快** | 无 |

## 理论关系

```
偏好优化方法演进:

RLHF (PPO)
  │
  ├── 去除奖励模型 → DPO (直接偏好优化)
  │     │
  │     ├── 放松配对要求 → KTO (二元标注)
  │     │
  │     └── 合并SFT+对齐 → ORPO
  │
  └── 去除标注数据 → GRPO (规则奖励)
```

### 数学等价性

DPO 和 RLHF 在理论上等价：
- DPO 的最优策略 = RLHF 的最优策略
- DPO 隐式学习了一个奖励函数
- 区别在于优化路径不同

KTO 和 DPO 的关系：
- KTO 使用前景理论替代 Bradley-Terry 模型
- KTO 的损失函数是 DPO 的推广
- 当所有数据都有配对时，KTO ≈ DPO

## 选择指南

### 按数据条件选择

| 你有什么数据 | 推荐方法 |
|-------------|----------|
| 成对偏好 + 预算充足 | RLHF/PPO |
| 成对偏好 + 预算有限 | **DPO** |
| 只有好/坏标注 | **KTO** |
| 只有任务提示 + 规则 | **GRPO** |
| 成对偏好 + 想省步骤 | ORPO |

### 按任务类型选择

| 任务类型 | 推荐方法 | 理由 |
|----------|----------|------|
| 通用对话 | DPO | 平衡效果和成本 |
| 数学/代码推理 | **GRPO** | 规则奖励+涌现能力 |
| 内容安全 | RLHF | 精细控制 |
| 快速对齐 | KTO | 数据获取容易 |
| 领域适配 | DPO/KTO | 不需要奖励模型 |

### 按资源条件选择

| 资源 | 推荐方法 |
|------|----------|
| 多卡 A100 集群 | RLHF/PPO (充分利用) |
| 单卡 24GB | DPO/KTO |
| 单卡 12GB | QLoRA + DPO |
| 极少标注预算 | GRPO (规则奖励) |

## 组合策略

### 经典管线：SFT → DPO
```
1. SFT (指令微调) → 学会回答格式
2. DPO (偏好优化) → 学会回答质量
```
最常用的开源对齐方案。

### DeepSeek-R1 管线：GRPO → SFT → DPO
```
1. GRPO → 涌现推理能力
2. SFT  → 规范输出格式
3. DPO  → 对齐人类偏好
```

### 高效管线：QLoRA + KTO
```
1. QLoRA SFT → 低显存微调
2. KTO → 只需二元标注
```
